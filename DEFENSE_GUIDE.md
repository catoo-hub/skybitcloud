# Skybit Cloud — конспект для защиты проекта

## 1) Коротко о проекте (что это и зачем)

Это MVP-платформа хостинга на Laravel, где есть:

- лендинг;
- регистрация и вход;
- роли `user/admin`;
- админка для управления товарами (тарифами);
- покупка сервера с настройками;
- пользовательская панель и заказы;
- управление «машиной» (включить/выключить, сменить пароль, переустановка, удаление).

Идея MVP: показать полный бизнес-цикл от витрины до покупки и дальнейшего управления услугой.

---

## 2) Почему именно Laravel + Blade + Tailwind

### Laravel

Я выбрал Laravel, потому что он закрывает весь базовый веб-стек из коробки:

- роутинг;
- ORM (Eloquent);
- миграции;
- сессии и аутентификация;
- валидация запросов;
- middleware;
- удобная структура MVC.

Для учебного/демо-проекта это быстро и предсказуемо.

### Blade

Blade-шаблоны позволяют быстро делать серверный рендеринг без лишней сложности фронтенд-фреймворков.

### Tailwind

Tailwind — utility-first CSS фреймворк: стили задаются маленькими классами прямо в разметке.
Почему это удобно в этом проекте:

- быстрая сборка интерфейса;
- единый стиль без длинных CSS-файлов;
- адаптивность без отдельной верстки;
- легко читать и править UI «на месте».

## 3) Что такое MVC и как это реализовано здесь

## MVC

- **Model** — работа с данными (таблицы БД, связи, бизнес-состояния).
- **View** — отображение (Blade-шаблоны).
- **Controller** — логика между запросом и ответом (валидация, вызов моделей, выбор view/redirect).

## В этом проекте

### Models

- `User` — пользователи, роль админа, связь с покупками.
- `Product` — тарифы/товары (цена, ресурсы, активность).
- `ServerPurchase` — купленные серверы и состояние машины.

### Views

- Лэндинг: `resources/views/welcome.blade.php`
- Auth: `resources/views/auth/*`
- Dashboard: `resources/views/dashboard/index.blade.php`
- Orders: `resources/views/orders/index.blade.php`, `resources/views/orders/manage.blade.php`
- Admin: `resources/views/admin/products/*`

### Controllers

- `AuthController` — регистрация/вход/выход.
- `HomeController` — главная/витрина.
- `DashboardController` — панель и список заказов.
- `PurchaseController` — checkout и покупка.
- `ServerMachineController` — управление машиной после покупки.
- `Admin\ProductController` — CRUD операций для товаров (в этом MVP create/update/delete + index/edit).

## 4) Роутинг: как идёт запрос от браузера

Пример цепочки:

1. Пользователь открывает `/dashboard`.
2. `routes/web.php` отправляет запрос в `DashboardController@index`.
3. Контроллер получает данные через модели (`Product`, `ServerPurchase`).
4. Передает данные в Blade-шаблон `dashboard.index`.
5. Blade рендерит HTML и отправляет в браузер.

Для защищенных роутов используется `middleware('auth')`.
Для админки дополнительно `middleware('admin')`.

## 5) База данных и миграции

## Что такое миграции

Миграции — это версионированное описание структуры БД в коде.

Плюсы:

- структура БД воспроизводима на любом окружении;
- изменения фиксируются в git;
- откат (`down`) и повторное применение (`up`).

## Какие таблицы используются

- `users` (+ поле `is_admin`)
- `products`
- `server_purchases`

## Важные поля `server_purchases`

- сервисные данные: `server_name`, `ip_address`, `port`, `username`, `password`
- конфигурация: `os`, `preset_software`
- состояние: `status`, `power_state`
- жизненный цикл: `expires_at`, `last_reinstalled_at`

## Команды, которые нужно знать на защите

```bash
php artisan migrate
php artisan migrate:rollback
php artisan migrate:fresh --seed
```

## 6) Авторизация и хэширование паролей

## Как сделано

- Регистрация через `AuthController@register`.
- Вход через `AuthController@login`.
- Выход через `AuthController@logout`.

Пароли хэшируются автоматически через cast `'password' => 'hashed'` в модели `User`.
Это значит:

- пароль никогда не хранится в открытом виде в БД;
- проверка при входе идет через безопасный механизм `Auth::attempt(...)`.

## Почему это важно

- защита пользовательских данных;
- базовое соответствие best practices по безопасности.

## 7) Роли и доступы (user/admin)

Добавлено поле `is_admin`.

### Логика доступа

- `auth` middleware — пускает только авторизованных.
- `admin` middleware — пускает только администраторов.

### Где используется

- `/admin/*` доступно только админам.
- обычные пользователи получают `403` при попытке зайти в админ-зону.

## 8) Товары и админка

Админ может:

- добавить товар;
- редактировать товар;
- удалить товар;
- управлять активностью товара (`is_active`).

Зачем это нужно:

- отделить управление тарифами от пользовательского интерфейса;
- админ меняет коммерческую часть, пользователи только покупают.

---

## 9) Покупка сервера (checkout)

Сейчас UX сделан в 2 шага:

1. На дашборде в карточке товара — только кнопка «Купить сервер».
2. Открывается отдельная страница checkout, где пользователь задаёт:
   - название сервера;
   - ОС;
   - предустановленное ПО;
   - пароль (сгенерировать или ввести вручную).

Почему так лучше, чем форма в карточке:

- меньше визуального шума;
- проще объяснить flow;
- меньше ошибок при вводе;
- масштабируется под дополнительные опции в будущем.

---

## 10) Управление машиной после покупки

В заказах теперь:

- краткая карточка сервера;
- статус с **индикатором-точкой**:
  - зеленая — включен (`running`),
  - красная — выключен (`stopped`);
- кнопка «Управление сервером».

На отдельной странице управления (`orders/manage`) доступны операции:

- включить;
- выключить;
- сменить пароль;
- переустановить сервер (ОС + ПО + пароль);
- удалить машину.

Почему отдельная страница управления лучше:

- список заказов остаётся компактным;
- все действия в одном понятном месте;
- меньше случайных кликов в списке.

## 11) Валидация и безопасность

Что применено:

- валидация входных данных в контроллерах (`$request->validate(...)`);
- CSRF-защита для форм (`@csrf`);
- проверка владельца ресурса в `ServerMachineController`:
  - пользователь может управлять только своими покупками,
  - админ может управлять любыми;
- `auth` и `admin` middleware для ограничения доступа.

---

## 12) Почему такие решения — аргументы для защиты

1. **MVP-подход**: реализован полный цикл, без избыточной сложности.
2. **Чёткая архитектура MVC**: логика отделена от отображения и БД.
3. **Безопасность по умолчанию**: хэш пароля, валидация, csrf, контроль доступа.
4. **Масштабируемый UX**: checkout и machine-manage вынесены в отдельные экраны.
5. **Админ/пользователь разделены**: бизнес-операции и клиентский интерфейс не смешаны.

## 13) Как объяснить жизненный цикл «покупки» на защите

Сценарий:

1. Пользователь логинится.
2. Заходит в дашборд и выбирает тариф.
3. Переходит на checkout и указывает параметры.
4. Нажимает «Подтвердить покупку».
5. Создаётся запись в `server_purchases` с фейковым доступом.
6. Заказ появляется в `Мои заказы`.
7. Через `Управление сервером` выполняются действия питания/пароля/переустановки/удаления.

## 14) Быстрые ответы на частые вопросы комиссии

### Почему не SPA на React/Vue?

Для MVP быстрее и надёжнее SSR на Blade: меньше инфраструктурной сложности, быстрее демонстрировать бизнес-логику.

### Почему пароль сервера хранится в открытом виде?

Это учебный «бутафорный доступ». Для продакшна хранить в открытом виде нельзя — нужен шифрованный storage или внешний секрет-менеджер.

### Почему статусы `active/inactive` и `running/stopped` одновременно?

`status` — общий бизнес-статус услуги, `power_state` — техническое состояние питания машины.

### Почему Tailwind, а не обычный CSS?

Tailwind ускоряет разработку интерфейса и делает UI-консистентность проще в MVP.

## 16) Короткий тезис для финала защиты

Я реализовал безопасный и расширяемый MVP хостинг-платформы на Laravel по MVC: от авторизации и ролей до покупки тарифа и управления сервером. Архитектурно разделил публичную витрину, checkout, пользовательскую панель и админку, а ключевые операции вынес в контроллеры с валидацией и ограничением доступа. Это даёт понятный фундамент для дальнейшего перехода от «бутафорного» доступа к реальному provisioning.
